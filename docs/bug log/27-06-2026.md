# Bug Log

## `/models` printed a libSQL ERROR and stalled "for a few secs" (synced/plain replica ping-pong)

**Symptom:** Opening `/models` printed `ERROR libsql::sync: local state is incorrect, db file exists but metadata file does not` then `Loading available models...`, hung a few seconds, and loaded fine. Looked `/models`-specific.

### Root cause

Not `/models`-specific and not a model-fetch delay — a stale-replica **re-bootstrap**. Sync tokens (`syncUrl`/`authToken`) reach `readDbConfig` only via Doppler env injection at startup (`tryInjectDoppler` in `index.ts`); there's no `~/.config/freecode/config.json` and the vars aren't in the shell. So `.freecode/freecode.db` is opened in **two** modes depending on whether Doppler ran:

- **With** Doppler → synced embedded replica (`createClient({ url, syncUrl, authToken })`).
- **Without** Doppler (direct `node dist/index.js`, pty test runs, Doppler absent) → **plain** local client (`createClient({ url })`) on the *same file*.

A plain write dirties the WAL and invalidates the replica's `-info` sync metadata. The next tokened run then sees inconsistent local state → libSQL logs the ERROR, discards the local replica, and **re-pulls the whole DB** (the delay). The live store's `freecode.db-info` read `"generation":15` — ~15 re-bootstraps, the smoking gun. `/models` is just where it's noticed: most-opened menu, and write-heavy (`persistModelRowAsync`), so a no-Doppler `/models` is what dirties the file for the next synced run. Suppressing the libSQL log (`RUST_LOG`) was rejected: cosmetic only — it hides the line but keeps the re-pull delay (and the binding isn't confirmed to honor `RUST_LOG`).

### Fix

`openAndPrepareClient` (`src/providers/db.ts`): in the no-tokens branch, if the db file is a sync replica (its `-info` sidecar exists), **decline to open it** — leave `client` null and let `doInit` degrade to an empty in-memory cache (no persistence that run), preserving the replica for the next sync. A genuinely tokenless user with no `-info` still opens a plain local file as before. Result: no plain-mode corruption, so no re-bootstrap, no delay, no ERROR line on the next tokened run.

### Coverage

- `tests/providers/db.test.ts`: new test plants `freecode.db` + a `freecode.db-info` sidecar with no tokens and asserts `initStore()` resolves to an empty cache **and leaves the sidecar byte-for-byte intact** (declined, not opened, not wiped). A second new test asserts the no-`-info` tokenless case still opens a real plain client (raw `SELECT 1` succeeds).
