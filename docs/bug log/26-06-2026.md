# Bug Log

## WalConflict crashed every slash-command menu (diverged libSQL replica + non-resilient init)

**Symptom:** Opening any menu (`/model`, `/config`, eval menu, …) printed `ERROR libsql::sync: insert error (frame=84): WalConflict` followed by an uncaught `Error: WalConflict` stack ending at `runMenuShell (dist/cli/menu-shell.js)`, with the bottom-UI footer interleaved into the trace. It recurred on *every* menu open.

### Root cause

Two compounding issues in `src/providers/db.ts`:

1. **A diverged embedded replica.** With Turso sync configured, the local replica's WAL diverged from the remote. `sync()` can't push the conflicting frames (the `frame=84 WalConflict` log lines), and — the part that wasn't handled — the *next write* throws too. `doInit` wrapped `client.sync()` in try/catch ("continuing offline") but left `createSchema()` unguarded, so its first `CREATE TABLE` (`Statement.run` → `executeStmt` in the trace) threw `WalConflict` uncaught.
2. **A non-resilient, memoized init.** `initStore()` does `initPromise ??= doInit()`. When `doInit` rejected, the rejected promise was cached, so every later `ensureStoreReady()` — which every menu awaits via `runMenuShell`'s `ensureReady` — re-threw the same error. Hence "every slash command."

### Fix

- `doInit` now **never throws**: all init work is wrapped; on failure it closes the bad client and degrades to an empty in-memory cache (reads empty, writes no-op), leaving the boot-primed config cache (`config-cache.json`) intact. A degraded init can't poison the memoized promise.
- **WalConflict recovery:** `isReplicaConflict(err)` gates a destructive recovery — close client, `wipeLocalDb()`, reopen, re-pull once. Transient network/auth sync errors keep the local replica and run offline (never wiped).
- **Correct wipe set:** the recovery wipe must delete the `-info` sidecar (libSQL replica sync metadata) or the conflict survives the re-pull. Verified against a real store dir: files are `freecode.db{,-shm,-wal,-info}` — there is no `-meta`. The pre-existing wipe list (`'', -shm, -wal, -meta`) was wrong for this case; `DB_FILE_SUFFIXES` now covers `'', -shm, -wal, -info, -meta`.

The user's live diverged replica self-heals on the next launch (wipe + re-pull). If divergence recurs, suspect a systematic cause (e.g. two processes writing the same replica) — re-pull heals it once but won't prevent a repeat.

### Coverage

- `tests/providers/db.test.ts`: new test poisons the db path (makes `freecode.db` a directory) and asserts `initStore()` **resolves** to an empty cache and a second call also resolves — i.e. the degrade path fires and the memoized promise isn't poisoned. WalConflict recovery itself is best-effort and left untested (hard to reproduce a real diverged replica deterministically; the destructive path is gated behind `isReplicaConflict`).

## Raw control bytes in a scenario JSON file broke JSON.parse (mangled escape sequence)

**Symptom:** `tty-config-parsed-tools.scenario.json` failed every run with `SyntaxError: Bad control character in string literal in JSON` during `docs:generate`/scenario load. It looked like the arrow-key sends had been written as bare `[A`/`[C`, but they had not.

### Root cause

Tool-call arguments are themselves transported as JSON. When a `Write` `content` parameter contains the 6-character text ``, the transport layer JSON-parses that escape into a real `0x1B` ESC byte *before* the `Write` tool receives the string, so a raw control byte gets written into the file. A raw control byte is illegal inside a JSON string literal, so the later `JSON.parse` throws. The file had three raw `0x1B` (ESC, from `[A`/`[C`) and one raw `0x03` (Ctrl-C, from the `"exit"` value). The working sibling `tty-config-load-agents-md.scenario.json` shows the correct form: literal text `\ u 0 0 1 b` on disk (verify with `od -c`).

### Fix

Replaced each raw control byte with its literal 6-char `\uXXXX` escape text, written via a Node script run in Bash (where bytes are literal and not transport-escaped) rather than through the `Write` tool. Verified with `od -c` (no `033`/`003`) and `JSON.parse`.

### Coverage (root-cause guard)

- New `tests/scenarios/scenario-json-bytes.test.ts`: scans every `tests/scenarios/*.json` for raw control bytes (allowing only tab/LF/CR) and fails with the file name, line, offending byte, and the fix. Converts the cryptic parse error into an actionable one, and runs inside `npm test`. Verified it fails on a mangled probe file, not just passes vacuously.
- `tests/harness/run-scenarios.ts`: the scenario-load `JSON.parse` is now wrapped to name the offending file instead of crashing anonymously, for direct `verify:scenarios` runs.

## Pre-tool-call preamble printed in the wrong place (glued onto the final step's text)

**Symptom:** After a multi-step turn, the model's pre-tool-call preamble appeared at the *end*, fused to the final response with no separating space — e.g. `Let's make this change.The script now runs successfully…`. The missing space was the tell that two text fragments generated at different times were concatenated by code, not by the model. Most visible on providers like Mistral that emit `content` alongside `tool_calls`.

### Root cause

Two compounding issues in `src/agent/loop.ts`:

1. **Markdown line buffer carried across the step boundary.** `streamWithRetry` creates one `createMarkdownStreamRenderer()` for the whole multi-step `textStream`. Its `lineBuffer` only emits on `\n` (`cli/markdown-renderer.ts`). A preamble like `"Let's make this change."` has no trailing newline (the model stopped mid-line to call a tool), so it sat buffered across the tool calls and was flushed only when the next step's text arrived — printing it in the wrong position, glued to the final text.
2. **`fullText` accumulation dropped the step boundary.** `fullText += chunk` joined every step's text with no separator, so the returned/stored text glued the same way.

Mistral's "streaming disabled" path is a red herring: `mistralForcedNonStream` only strips `stream:true` at the HTTP layer, then `mistralJsonToSse` re-synthesizes SSE through the *same* AI SDK multi-step pipeline, so `loop.ts` accumulated text across steps identically.

### Fix

At each tool-call step boundary, flush the pending partial line so the preamble lands in its correct position (after this step's text, before the tool calls):

- Native streaming path: in `onStepFinish` (on `finishReason === 'tool-calls'`), push a `\n` through the markdown renderer to stdout/transcript and append `\n` to `fullText`.
- Fake-direct (`mock`) path: append `\n` to `fullText` after `executeToolCalls` resolves (not on abort — an aborted call has no next step), and have `runFakeModel` (`src/providers/fake.ts`) terminate each step's stdout text with a newline.

### Coverage

- New scenario `agent-preamble-flush` (`tests/scenarios/`): preamble with no trailing newline + a tool call, asserting `stdoutAbsent: ["change.The script"]` and that both lines appear separately.
- `tests/agent/loop.test.ts` assertions updated from the old glued strings (e.g. `Writing now.Finished.` → `Writing now.\nFinished.`); the abort test keeps `'Thinking. '` unchanged.
